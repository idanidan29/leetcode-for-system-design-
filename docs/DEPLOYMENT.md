# Deployment guide

How to put sketchd on the internet. Free tier on every service.

```
   Browser
      │
      ▼
  Vercel  ───────────────►  Render  ───────────────►  Neon
  (Next.js)                 (FastAPI + uvicorn)        (Postgres)
                                  │
                                  ▼
                                Groq
                                (LLM, free tier)
```

Total moving parts: 3 services + 1 API key. Estimated time end-to-end: 30–45 minutes the first time.

---

## Pre-flight

1. Code is pushed to a public GitHub repo (Vercel and Render both pull from GitHub).
2. You have accounts on:
   - [Vercel](https://vercel.com) — sign in with GitHub.
   - [Render](https://render.com) — sign in with GitHub.
   - [Neon](https://neon.tech) — free Postgres, no card required.
   - [Groq](https://console.groq.com) — already have one if you've been running locally.

---

## Step 1 — Create the database (Neon)

1. Sign in at [console.neon.tech](https://console.neon.tech) and click **Create project**.
2. Region: pick one near where you'll deploy Render (`Oregon (us-west-2)` if you keep Render's default). Same region = lower latency.
3. Postgres version: 16 (default).
4. After creation, Neon shows a **Connection string** that looks like:
   ```
   postgresql://my_user:my_pass@ep-cool-name-12345.us-west-2.aws.neon.tech/neondb?sslmode=require
   ```
   Copy it — you'll paste it into Render as `DATABASE_URL`.

> The app auto-normalizes this to the asyncpg form (`postgresql+asyncpg://...?ssl=require`) so you can paste Neon's string as-is.

---

## Step 2 — Deploy the API (Render)

The repo already includes [`apps/api/render.yaml`](../apps/api/render.yaml) — a Render Blueprint that describes the service. Easiest path:

1. In Render, click **New → Blueprint**.
2. Connect your GitHub repo and pick this one.
3. Render reads `render.yaml` and shows you a single service: `sketchd-api`.
4. Fill in the three secrets Render leaves blank:

   | Env var | Value |
   |---|---|
   | `DATABASE_URL` | the Neon connection string from Step 1 |
   | `GROQ_API_KEY` | your key from console.groq.com |
   | `CORS_ORIGINS` | leave blank for now — we'll fill it after Step 3 |

   `JWT_SECRET` is auto-generated by Render. `COOKIE_SECURE=true` and `COOKIE_SAMESITE=none` come from the blueprint.

5. Click **Apply**. Render will:
   - Install Python deps (`pip install -e .`).
   - Run **`alembic upgrade head`** as the pre-deploy step. This creates all tables on the Neon database.
   - Start uvicorn.

6. Once the service is **Live**, copy its URL — something like `https://sketchd-api-abc.onrender.com`. Health-check it: `curl https://<your-service>.onrender.com/health` should return `{"status":"ok"}`.

### Seed the catalog (one-shot)

`render.yaml` runs migrations on every deploy but doesn't seed problems (intentionally — seeding is mutative and shouldn't be automatic). From the **Render dashboard → Shell** tab:

```bash
python scripts/seed_problems.py
```

You should see "23 problems in the catalog" with a system-design / design-patterns grouping.

> Free tier services on Render **sleep after 15 minutes of inactivity**. The first request after sleep takes 30–60 seconds to wake. This is fine for a portfolio project but worth knowing — you'll want to mention it in your README.

---

## Step 3 — Deploy the frontend (Vercel)

1. In Vercel, click **Add New → Project**.
2. Import the same GitHub repo.
3. Configure the project:
   - **Root Directory**: `apps/web`
   - **Framework Preset**: Next.js (auto-detected)
   - **Environment Variables**: add one
     - `NEXT_PUBLIC_API_URL` = `https://<your-render-service>.onrender.com/api/v1`
4. Deploy.

After the build finishes, Vercel hands you a URL like `https://sketchd-username.vercel.app`. Open it.

---

## Step 4 — Plug Vercel and Render together

The frontend is talking to the backend, but cookies will fail because CORS still only allows localhost. Fix in Render:

1. In the `sketchd-api` service → **Environment** → edit `CORS_ORIGINS`:
   ```
   https://sketchd-username.vercel.app
   ```
   (No quotes, no brackets — just a comma-separated list of origins. The app accepts CSV.)
2. Save. Render restarts the service automatically (~30s).
3. In your browser, sign up on the Vercel URL. The httpOnly session cookie should land, and `/profile` should load.

### Custom domain (optional, recommended)

If you have a domain:

- **Vercel**: Project → Settings → Domains → add `sketchd.app` (or whatever). Update DNS as instructed.
- **Render**: Service → Settings → Custom Domains → add `api.sketchd.app`. Update DNS.
- Update env vars:
  - Render `CORS_ORIGINS`: `https://sketchd.app`
  - Render `COOKIE_DOMAIN`: `.sketchd.app` (note the leading dot — covers all subdomains)
  - Render `COOKIE_SAMESITE`: `lax` (you can drop `none` when frontend + API share a parent domain — strictly nicer for security)
  - Vercel `NEXT_PUBLIC_API_URL`: `https://api.sketchd.app/api/v1`
- Redeploy both.

## Things that will probably bite you

**1. Cookies aren't sticking after login.**
Open DevTools → Application → Cookies. If the `access_token` cookie isn't there:
- `COOKIE_SECURE=true` requires HTTPS. Check the Render service URL is `https://`.
- `COOKIE_SAMESITE=none` requires `COOKIE_SECURE=true`. Both must be on for cross-site (Vercel↔Render).
- `CORS_ORIGINS` must include the exact Vercel origin, no trailing slash.
- The frontend's `fetch` must include `credentials: "include"` — it already does, but if you forked and changed, check `apps/web/lib/api.ts`.

**2. `/problems` returns 500 right after deploy.**
The pre-deploy migration didn't run. Check Render → Events. If it failed, run manually from the Shell:
```bash
alembic upgrade head
```

**3. CORS preflight fails (`OPTIONS` returns 405 or no headers).**
The CORS middleware is wired before the routes — that part is correct. The most common cause is `CORS_ORIGINS` set wrong: the value must match `window.location.origin` exactly (scheme + host + port, no path).

**4. Render free tier sleeping causes the first request to time out.**
Two fixes:
- Live with it — note it on the README.
- Wire a free-tier uptime monitor (Better Stack, UptimeRobot) to ping `/health` every 5 minutes. This keeps the dyno warm but uses your monthly free hours faster.

**5. Neon free tier auto-suspends the database too.**
Neon's free compute is "scale to zero" after 5 minutes idle. First query after suspension takes a few hundred ms to wake — usually invisible, but worth knowing.

## Updating after the first deploy

- **Code changes**: push to `main` on GitHub. Both Vercel and Render auto-deploy from the connected branch.
- **New problems**: edit `apps/api/scripts/problems/system_design.py` or `design_patterns.py`, push, then run `python scripts/seed_problems.py` from the Render Shell.
- **New migration**: include it in your PR. Render runs `alembic upgrade head` automatically on the next deploy.
- **Env var change**: edit in Render or Vercel dashboard. Both restart the service automatically.

---

## Cost

Everything in this guide is free-tier indefinitely *unless* one of:

- You exceed 100 GB egress / month on Vercel.
- You exceed 750 hours / month on Render (one service running 24/7 = 720 hours).
- You exceed 0.5 GB on Neon's free Postgres.

For a portfolio project, you won't come close. If it goes viral, the limiting factor will be Groq's free tier (1k requests/day on Llama 3.3 70B) — at which point your problem is good.
